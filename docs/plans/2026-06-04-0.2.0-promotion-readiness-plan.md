# 0.2.0 Promotion Readiness Plan

[Korean](../ko/plans/2026-06-04-0.2.0-promotion-readiness-plan.md)

Status: first audit slice implemented

Codexus should not treat `0.2.0` as the version that happens after `0.1.9`.
The existing JSON contract defines `0.2.0` as the point where Codexus either
promotes one or more experimental evidence surfaces into the stable contract, or
intentionally changes already frozen stable fields with changelog notice.

This plan is a guardrail, not a schedule commitment. It does not require the
project to cut `0.2.0` soon. It exists so that, if a maintainer considers
`0.2.0`, the decision is tied to stable-contract evidence instead of version
number intuition.

## Bottom Line

1. `0.2.0` is a contract milestone, not a countdown.
2. Broad 0.2/0.3 feature completion is not required for `0.2.0`.
3. At least one low-risk experimental surface should be promoted and frozen
   before cutting `0.2.0`.
4. If no surface is ready, continue the `0.1.x` line (`0.1.10`, `0.1.11`, and
   so on) instead of forcing `0.2.0`.
5. Action surfaces such as app-instance start/stop, live autopilot, active relay
   adapters, Desktop attachment, automatic injection, and plugin always-on claims
   remain experimental/deferred until their lifecycle and authority evidence is
   stronger.

## Implemented Audit Surface

`cx contract check --json` now reports a structured 0.2.0 promotion audit:

```bash
cx contract check --target 0.2.0 --json
cx contract check --target 0.2.0 --gate --json
```

The default report is informational. `--gate` fails until at least one audited
candidate is actually stable and its frozen fields are documented in
`docs/json-contract.md`. This failure is intentional: it prevents a release
manager from mistaking "candidate exists" for "0.2.0 is ready."

## Candidate Set

The first audit slice identifies these viable candidates:

- `repo check --gate`: strongest candidate. It is local, static, dogfooded by the
  repository knowledge gate, and does not execute external tools.
- `release check --gate`: strong candidate once local mode and live mode have
  separately frozen fields. It already protects trusted publishing and installer
  integrity.
- `lsp check --gate`: possible candidate after more dogfooding. It runs only an
  explicit diagnostics command and starts no protocol server, but non-TypeScript
  and unavailable branches need stable contract coverage.
- `architecture check --gate`: possible candidate for the narrow
  declared-policy forbidden-import subset only. Broad layering remains deferred.

## Explicit Non-Candidates

The first `0.2.0` promotion should not include:

- `app instance start|stop`
- live `autopilot run`
- active multi-engine relay adapters
- Desktop app-server live attachment
- automatic context injection
- plugin-packaging always-on claims

These surfaces either have real process/control side effects, lack an upstream
supported bridge, or are intentionally distribution/discoverability experiments.

## Promotion Procedure

1. Pick one or two candidates, preferably `repo check` and/or local-mode
   `release check`.
2. Freeze their named top-level JSON fields in `docs/json-contract.md` and the
   Korean counterpart.
3. Add contract tests proving unknown additive fields remain allowed but named
   fields are not removed or redefined.
4. Add installed-package smoke for the promoted commands.
5. Update implementation status, remaining work, and the kanban board so
   promoted surfaces are marked stable and non-promoted surfaces remain
   experimental/deferred.
6. When intentionally evaluating a `0.2.0` cut, run
   `cx contract check --target 0.2.0 --gate --json`. It should pass only after
   a stable promotion exists.
7. Cut `0.2.0` with release notes that name the promoted surfaces and explicitly
   state which 0.2/0.3 tracks remain experimental.

## Current Result

As of this first slice, Codexus is **not ready** for `0.2.0` because all audited
promotion candidates still self-report as non-stable. That is the correct
result. The next release can continue the `0.1.x` line indefinitely unless a
promotion hardening slice is intentionally completed.
