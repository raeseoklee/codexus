# 0.1.0 Stable Readiness Plan

[Korean](../ko/plans/2026-05-31-0.1.0-stable-readiness-plan.md)

Date: 2026-05-31
Status: active readiness plan
Current baseline: `0.1.0-alpha.4` published to npm (`latest = next = 0.1.0-alpha.4`)

## Purpose

This plan defines the cut from `0.1.0-alpha.4` to **`0.1.0` stable**. It is the
missing middle between [remaining-work.md](../remaining-work.md) (the long-term
product backlog) and [public-release.md](../public-release.md) (the
open-source/alpha publish checklist). It answers one question: **what will
Codexus promise as stable, and what will it explicitly not promise yet.**

The guiding principle is the project's own discipline: *the priority is not
shipping more features, it is making sure nothing that does not work looks like
it works.* 0.1.0 is a subtraction exercise, not an addition one.

## Bottom line

1. **0.1.0 stable** = latest-installable + stable JSON contract frozen through 0.1.x.
2. **0.1.0 publish** = GitHub Actions trusted publishing; local token publish is fallback only.
3. **Release blockers** = derivable evidence only.
4. **Manual E2E** = a redacted sign-off artifact (`docs/release-evidence/0.1.0.md`).
5. **Experimental surfaces** must self-report as `experimental` / `deferred`.

## What "stable" means here

**Definition:** 0.1.0 stable is *the first build you can install from `latest`,
with the stable-surface JSON contract frozen through 0.1.x* — not "all features
complete." The freeze covers the **stable surface only**; experimental/deferred
surfaces stay outside the freeze.

0.1.0 is **a contract-frozen 0.1.x line**, not 1.0.

- 0.x semver still allows breaking changes in a later **minor** (0.2.0). 0.1.0
  does **not** promise long-term API stability.
- Within the **0.1.x patch line**, the JSON output contract of *supported*
  commands is **frozen**: named fields are not removed or redefined; only
  additive fields may appear.
- Breaking changes to frozen fields require **0.2.0**.
- Experimental command output is **not** covered by the freeze and is labeled as
  such (see "JSON contract freeze").

This is the (b) interpretation chosen on 2026-05-31: automation can rely on
supported-command JSON across 0.1.x.

## Supported surface (the stable contract)

The stable engine boundary is the local authenticated **`codex exec --json`**
path and the local-evidence surfaces built on it. The following are supported
and contract-frozen in 0.1.0:

- Core run loop: `doctor`, `init`, `run`, `cancel`, `plan`, `status`,
  `runs list`, `events tail`, `report`, `resume`, `verify`.
- Drivers: `mock` (deterministic) and `codex-exec` (local authenticated Codex).
- Replay/integrity: `replay`, `replay parity`, `locks list|inspect|clear`.
- Schema: `schema check`, `schema engine`, `schema validate`,
  `schema validate-run` (local subset engine; full JSON Schema engine stays
  out of scope).
- Memory lifecycle: `memory add|search|list|review|curate|prune`.
- Skill lifecycle: `skill propose|index|list|review|promote|export|improve|
  deprecate` (replay-gated; export is explicit, never auto-install).
- Quality evidence: `slop check` / `session slop` (tri-state, `--gate`).
- Supply-chain evidence: `supply-chain check` (`--json` / `--gate`,
  static projection, no lifecycle execution).
- Codex-native session surface (read/record/derive only): `session status`,
  `session checkpoint`, `session verify` (`--auto` detect, `--execute` gated),
  `session hud`, `session migrate`, `session subagent record|attach|status`
  (recorder-only), `session workers status` (gate report only),
  `session notify` (internal hook write surface).
- Setup/adapter: `setup codex-session` (`--always-on`, `--enable-notify-hook`,
  trust-gated, chain-preserving) and the installed `$codexus` Codex skill
  adapter.
- Packaging: npm global install of `cx`/`codexus` with `dist/cli/main.js`,
  postinstall skill adapter, and `install.sh`.

Rule for "supported": the command reports or derives **local evidence** and does
not depend on any live dispatch, injection, launch, or network database.

## Explicitly excluded / experimental in 0.1.0

These remain behind gates and are **promised as deferred**, not as working. They
must be visibly labeled experimental (see freeze section) and must never affect
the stable `codex exec --json` path:

- Live app-server turn execution (`app-server *` stays status/dry-run/record
  only; live execution gated off; Desktop attachment unobserved).
- Real cron/gateway dispatch (`cron *`, `gateway *` stay status/dry-run/audit
  only; share `policy-reviewed-live-dispatch-v1`; blocked until a dispatcher
  exists).
- Automatic prompt injection of retrieved skills (`adapt omx injection` records
  an approval artifact only; never auto-injects).
- Routine live model-in-the-loop replay (env/budget-gated; not default).
- Full external JSON Schema (draft-2020-12) engine enforcement/migrations.
- Statusline integration (unavailable; `session hud --json` is the fallback).
- tmux-backed worker launch (`session workers status` reports the gate; no
  launch).
- Native subagent launcher (recorder-only; record/attach/status are supported,
  spawn is not).

These move to the **0.2 / 0.3 track**, not into 0.1.0.

## JSON contract freeze

- **Frozen in 0.1.x:** the named top-level fields of `--json` output for every
  supported command above (e.g. the run ledger `state.json` core, `status`,
  `doctor`, `supply-chain check` report shape, `slop check` report shape,
  `session status`, `schema check`). Removing or redefining a frozen field is a
  0.2.0 change.
- **Additive only:** new fields may be added within 0.1.x; consumers must ignore
  unknown fields.
- **Not frozen:** experimental command output (app-server/cron/gateway/model
  replay/injection), and the *contents* of advisory lanes (`heuristicClaims`)
  whose membership is explicitly heuristic.
- **Self-describing stability (required for 0.1.0):** add a `"stability":
  "stable" | "experimental" | "deferred"` marker to command `--json` output so a
  consumer knows from the payload alone whether the contract applies. To keep the
  change bounded:
  - put it at the top level or under `capability.stability`, alongside any
    existing `schemaVersion`;
  - before 0.1.0, apply it to at least `doctor`, `session status`,
    `app-server status`, `cron status` / `gateway status`, `schema engine`, and
    `supply-chain check`;
  - remaining commands gain the marker as a 0.1.x additive follow-up, with the
    README support matrix as the interim source of truth.
- `docs/json-contract.md` (EN/KO) enumerates the frozen field set and the
  breaking-change policy.

## Release blockers (derivable pass criteria)

0.1.0 does not ship until each of these produces checkable evidence — no
subjective "looks polished":

1. `npm run release:check` green (local CI + package smoke + supply-chain
   report-only dogfood).
2. Full unit test suite + `npm run typecheck` green; GitHub CI green.
3. `supply-chain check --gate` passes on this repository.
4. Extended automated E2E (below) green against the **installed tarball**.
5. Manual sign-off checklist (below) complete with attached evidence.
6. Trusted-publishing workflow performs one provenance-bearing publish
   successfully.
7. Docs polish checklist complete.
8. Git release tag applied (`v0.1.0`) and pushed.

## E2E smoke matrix

Split by what CI can run unattended versus what needs a human and a live Codex.

### Automated (no model/network; extend `package-smoke`)

- `npm pack` → install into an isolated global prefix → `cx`/`codexus`
  `--version`, `--help`.
- Installed binary: `doctor --json --strict`, `schema check --json`,
  `supply-chain check --gate --json`. The automated `doctor --strict` path uses
  an isolated project config and fake `codex` fixture so CI never needs a live
  model login.
- Installed binary with `--driver mock`: `run` with passing verification,
  failing verification, and one repair iteration; then `status`, `resume`,
  `cancel`, `events tail`.
- postinstall Codex skill adapter installed under an isolated `CODEX_HOME`.

### Manual sign-off (attach evidence to the release)

- Real `codex-exec`: `codexus run --verify "npm test" "<task>"` against a tiny
  throwaway fixture repo; capture the terminal outcome and ledger id.
- `$codexus` skill flow once inside a Codex CLI/TUI session: `session status`,
  `session checkpoint`, `session verify`.
- `resume` and `cancel` exercised once each against real runs.
- One real notify-hook `turn-ended` observed (`capabilities.hooks` flips to
  `available`).

Record the manual results as a **redacted summary** in
`docs/release-evidence/0.1.0.md` (EN/KO) — never commit raw run ledgers or
transcripts.

## npm / publish / trusted publishing

Decision (2026-05-31): **migrate to npm trusted publishing (OIDC) with
provenance** before 0.1.0.

- Add a GitHub Actions release workflow with `permissions: id-token: write`,
  running on a **GitHub-hosted runner** (npm trusted publishing requires this),
  that publishes from CI. Trusted publishing generates **provenance
  automatically** — no `--provenance` token plumbing.
- Configure the npm package's trusted-publisher settings to match: repository,
  the exact **workflow filename**, and the allowed action/environment.
- Local `npm run publish:next` becomes a **fallback/dev path**, not the 0.1.0
  release path. Keep its `latest >= next` dist-tag guard.
- Prove the trusted-publishing path with a prerelease rehearsal, for example
  `0.1.0-alpha.5` via `workflow_dispatch mode=next`, before tagging `v0.1.0`.
  Tag-triggered publish is reserved for non-prerelease versions.
- After cutover, `publish_provenance` is satisfied **for Codexus's own
  releases** (it remains an `informationalUnknown` for arbitrary third-party
  packages the checker inspects — that is correct).
- Improve registry-propagation handling: the dist-tag verification step should
  retry with backoff instead of failing on read-after-write lag.
- Keep `npm_2fa_status` / `publish_provenance` documented as
  `informationalUnknown` for the general checker, with a note that Codexus's own
  pipeline resolves provenance via trusted publishing.

References: [npm trusted publishers](https://docs.npmjs.com/trusted-publishers/),
[npm trust CLI](https://docs.npmjs.com/cli/v11/commands/npm-trust/).

## 0.1.0 release procedure

1. Bump `package.json` to `0.1.0`; update `CHANGELOG.md` (EN/KO).
2. `npm run release:check` green.
3. Automated E2E matrix green against the packed tarball.
4. Manual sign-off checklist complete with evidence.
5. Verify npm trusted-publisher settings point at
   `raeseoklee/codexus/.github/workflows/release.yml`.
6. Apply and push `git tag v0.1.0`; the trusted-publishing workflow publishes
   from the tag with provenance attached.
7. Confirm `npm dist-tags` shows `latest = 0.1.0` and `next >= 0.1.0`.
8. Re-verify the installed `latest` tarball (`cx --version`, `doctor --strict`,
   `supply-chain check --gate`).

## Out of scope for 0.1.0 (0.2 / 0.3 track)

- Live app-server turn execution.
- Real cron/gateway dispatch.
- Automatic prompt injection.
- Full JSON Schema engine dependency.
- Statusline integration.
- tmux-backed worker launch.
- Routine live model replay.

## Open decisions

1. **Recorder-only session sub-surfaces.** Confirm `session subagent
   record|attach|status` and `session workers status` are promised as supported
   (report/record), with launch/spawn explicitly experimental.

## Suggested work order

`this readiness plan → docs/json-contract.md + README support matrix →
extend package-smoke into the automated E2E matrix → trusted-publishing
workflow + propagation backoff → docs polish → manual sign-off → 0.1.0 cut`.
