# 0.1.0 Stable Readiness Plan (정식 준비 계획)

[English](../../plans/2026-05-31-0.1.0-stable-readiness-plan.md)

작성일: 2026-05-31
상태: 활성 readiness 계획; prerelease evidence 수집 완료
현재 기준선: `0.1.0-alpha.5` npm 배포됨 (`latest = next = 0.1.0-alpha.5`)
현재 blocker: trusted-publishing rehearsal 통과 전 npm trusted-publisher 설정에
maintainer 2FA action이 필요합니다.

## 목적

이 계획은 `0.1.0-alpha.5`에서 **`0.1.0` stable**로 넘어가는 컷 기준을 정의합니다.
[remaining-work.md](../remaining-work.md)(장기 제품 백로그)와
[public-release.md](../public-release.md)(오픈소스 공개/alpha publish 체크리스트)
사이에 비어 있던 중간 단계입니다. 답하려는 질문은 하나입니다: **Codexus가 stable로
무엇을 약속하고, 무엇을 아직 약속하지 않을 것인가.**

지침 원칙은 이 프로젝트의 정체성 그대로입니다: *우선순위는 기능을 더 내는 게 아니라,
안 되는 것이 되는 것처럼 보이지 않게 하는 것.* 0.1.0은 더하기가 아니라 **빼기** 작업입니다.

## 핵심 결론

1. **0.1.0 stable** = latest-installable + stable JSON contract가 0.1.x 동안 frozen.
2. **0.1.0 publish** = GitHub Actions trusted publishing; 로컬 토큰 publish는 fallback 전용.
3. **Release blockers** = derivable 증거만.
4. **수동 E2E** = redacted sign-off artifact(`docs/release-evidence/0.1.0.md`).
5. **experimental surface**는 `experimental` / `deferred`로 자기보고해야 함.

## 여기서 "stable"의 의미

**정의:** 0.1.0 stable은 *`latest`로 설치해도 되는 첫 빌드이며, stable surface의 JSON
contract가 0.1.x 동안 깨지지 않는 것* 입니다 — "모든 기능 완성"이 아님. freeze는
**stable surface에만** 적용되고, experimental/deferred surface는 freeze 대상 밖입니다.

0.1.0은 **contract-frozen 0.1.x 라인**이지 1.0이 아닙니다.

- 0.x semver는 이후 **minor**(0.2.0)에서 breaking을 여전히 허용합니다. 0.1.0은 장기
  API 안정성을 약속하지 **않습니다**.
- **0.1.x 패치 라인 안에서는** *supported* 명령의 JSON 출력 contract가 **freeze**됩니다:
  named 필드는 제거·재정의되지 않고, additive 필드만 추가될 수 있습니다.
- frozen 필드의 breaking 변경은 **0.2.0**이 필요합니다.
- experimental 명령 출력은 freeze 대상이 **아니며** 그렇게 라벨됩니다("JSON contract
  freeze" 참조).

이는 2026-05-31에 선택한 (b) 해석입니다: 자동화가 0.1.x 전체에서 supported 명령 JSON을
신뢰할 수 있습니다.

## Supported surface (stable contract)

stable 엔진 경계는 로컬 인증된 **`codex exec --json`** 경로와 그 위에 만든 로컬 증거
surface입니다. 아래는 0.1.0에서 supported이며 contract-frozen입니다:

- 코어 run loop: `doctor`, `init`, `run`, `cancel`, `plan`, `status`,
  `runs list`, `events tail`, `report`, `resume`, `verify`.
- 드라이버: `mock`(deterministic), `codex-exec`(로컬 인증 Codex).
- replay/무결성: `replay`, `replay parity`, `locks list|inspect|clear`.
- 스키마: `schema check`, `schema engine`, `schema validate`,
  `schema validate-run` (로컬 subset 엔진; full JSON Schema 엔진은 범위 밖).
- 메모리 lifecycle: `memory add|search|list|review|curate|prune`.
- 스킬 lifecycle: `skill propose|index|list|review|promote|export|improve|
  deprecate` (replay-gated; export는 명시적, 자동 설치 안 함).
- 품질 증거: `slop check` / `session slop` (tri-state, `--gate`).
- 공급망 증거: `supply-chain check` (`--json` / `--gate`, 정적 projection,
  lifecycle 미실행).
- Codex-native 세션 surface (read/record/derive 전용): `session status`,
  `session checkpoint`, `session verify`(`--auto` 탐지, `--execute` gated),
  `session hud`, `session migrate`, `session subagent record|attach|status`
  (recorder-only), `session workers status`(gate 리포트 전용),
  `session notify`(내부 hook write surface).
- setup/adapter: `setup codex-session`(`--always-on`, `--enable-notify-hook`,
  trust-gated, chain 보존)과 설치된 `$codexus` Codex skill adapter.
- 패키징: `cx`/`codexus`의 npm 글로벌 설치(`dist/cli/main.js`), postinstall skill
  adapter, `install.sh`.

"supported" 기준: 명령이 **로컬 증거**를 보고·도출하며, live dispatch·injection·launch·
네트워크 DB에 의존하지 않을 것.

## 0.1.0에서 명시적으로 제외 / experimental

아래는 gate 뒤에 남으며 **작동이 아니라 deferred로 약속**됩니다. experimental로 가시적
라벨되어야 하고(freeze 섹션 참조), stable `codex exec --json` 경로에 영향을 줘선 안 됩니다:

- Live app-server turn 실행 (`app-server *`는 status/dry-run/record 전용; live 실행
  gated off; Desktop attachment 미관측).
- 실제 cron/gateway dispatch (`cron *`, `gateway *`는 status/dry-run/audit 전용;
  `policy-reviewed-live-dispatch-v1` 공유; dispatcher 생길 때까지 blocked).
- 검색된 skill의 자동 prompt injection (`adapt omx injection`은 approval artifact만
  기록; 자동 주입 안 함).
- routine live model-in-the-loop replay (env/budget-gated; 기본 아님).
- full 외부 JSON Schema(draft-2020-12) 엔진 강제/마이그레이션.
- statusline 통합 (불가; `session hud --json`이 fallback).
- tmux 기반 worker launch (`session workers status`가 gate 보고; launch 없음).
- 네이티브 subagent launcher (recorder-only; record/attach/status는 supported,
  spawn은 아님).

이들은 0.1.0이 아니라 **0.2 / 0.3 트랙**으로 갑니다.

## JSON contract freeze

- **0.1.x에서 frozen:** 위 모든 supported 명령의 `--json` 출력의 named top-level 필드
  (예: run ledger `state.json` 코어, `status`, `doctor`, `supply-chain check` 리포트
  형태, `slop check` 리포트 형태, `session status`, `schema check`). frozen 필드 제거·
  재정의는 0.2.0 변경.
- **additive only:** 0.1.x 안에서 새 필드 추가 가능; 소비자는 미지 필드를 무시해야 함.
- **frozen 아님:** experimental 명령 출력(app-server/cron/gateway/model replay/
  injection), 그리고 멤버십이 명시적으로 heuristic인 advisory lane(`heuristicClaims`)의
  *내용*.
- **자기서술 stability (0.1.0 필수):** 명령 `--json` 출력에 `"stability":
  "stable" | "experimental" | "deferred"` 마커를 추가해 소비자가 payload만으로 contract
  적용 여부를 알게 함. 범위를 한정하려면:
  - top-level 또는 `capability.stability`에 두고, 기존 `schemaVersion`과 함께 유지;
  - 모든 명령에 한 번에 넣지 말고, 0.1.0 전엔 최소 `doctor`, `session status`,
    `app-server status`, `cron status` / `gateway status`, `schema engine`,
    `supply-chain check`부터 적용;
  - 나머지 명령은 0.1.x additive 후속으로 마커 추가, 그동안 README support 매트릭스가
    잠정 진실 출처.
- `docs/json-contract.md`(EN/KO)가 frozen 필드 집합과 breaking-change 정책을 열거.

## Release blockers (derivable 통과 기준)

0.1.0은 각 항목이 체크 가능한 증거를 낼 때까지 배포하지 않습니다 — 주관적 "polish 됐다"
금지:

1. `npm run release:check` green (로컬 CI + package smoke + supply-chain report-only
   dogfood).
2. 전체 unit test + `npm run typecheck` green; GitHub CI green.
3. 이 repository에서 `supply-chain check --gate` 통과.
4. 확장된 자동 E2E(아래)가 **설치본 tarball** 기준 green.
5. 수동 sign-off 체크리스트(아래) 증거 첨부 완료.
6. trusted-publishing 워크플로가 provenance 부착 publish를 1회 성공. 현재 상태:
   npm trusted-publisher 설정에서 blocked.
7. docs polish 체크리스트 완료. 현재 상태: 0.1.0 cut 기준 완료.
8. git release 태그(`v0.1.0`) 적용·푸시.

## E2E smoke matrix

CI가 무인 실행할 수 있는 것과, 사람+실제 Codex가 필요한 것으로 분리.

### 자동 (모델/네트워크 불요; `package-smoke` 확장)

- `npm pack` → 격리 글로벌 prefix 설치 → `cx`/`codexus` `--version`, `--help`.
- 설치본: `doctor --json --strict`, `schema check --json`,
  `supply-chain check --gate --json`. 자동 `doctor --strict` 경로는 격리 project config와
  fake `codex` fixture를 사용하므로 CI에 live model login이 필요하지 않습니다.
- 설치본 `--driver mock`: verification 통과/실패/repair 1회 `run`; 이어서 `status`,
  `resume`, `cancel`, `events tail`.
- 격리 `CODEX_HOME`에 postinstall Codex skill adapter 설치 확인.

### 수동 sign-off (릴리스에 증거 첨부)

- 실제 `codex-exec`: 작은 throwaway fixture repo에 `codexus run --verify "npm test"
  "<task>"`; 터미널 결과와 ledger id 캡처. 현재 상태: prerelease evidence artifact
  기준 완료.
- Codex CLI/TUI 세션 안에서 `$codexus` skill 흐름 1회: `session status`,
  `session checkpoint`, `session verify`. 현재 상태: 완료.
- 실제 run에 `resume`, `cancel` 각 1회. 현재 상태: live cancel은 완료. 더 엄격한
  final tag 증거를 원하면 successful real resume completion을 한 번 더 반복할 수 있습니다.
- 실제 notify-hook `turn-ended` 1회 관측(`capabilities.hooks`가 `available`로 전환).
  현재 상태: 완료.

수동 결과는 **redacted summary**로 `docs/release-evidence/0.1.0.md`(EN/KO)에 기록 —
raw run ledger나 transcript는 절대 커밋하지 말 것.

## npm / publish / trusted publishing

결정(2026-05-31): 0.1.0 전에 **npm trusted publishing(OIDC) + provenance로 전환**.

- `permissions: id-token: write`를 가진 GitHub Actions 릴리스 워크플로를 **GitHub-hosted
  runner**에서 실행(npm trusted publishing 전제)하고 CI에서 publish. trusted publishing은
  **provenance를 자동 생성**(별도 `--provenance` 토큰 배선 불필요).
- npm 패키지의 trusted-publisher 설정을 맞춤: repository, 정확한 **workflow 파일명**,
  허용 action/environment.
- 로컬 `npm run publish:next`는 0.1.0 릴리스 경로가 아니라 **fallback/dev 경로**가 됨.
  `latest >= next` dist-tag 가드는 유지.
- `v0.1.0` 태깅 전 trusted-publishing 경로를 prerelease rehearsal로 증명합니다. 첫
  `0.1.0-alpha.5` workflow rehearsal은 npm trusted-publisher/2FA 설정 부재로
  blocked였습니다. Fallback token publish는 성공했지만 이 blocker를 충족하지 않습니다.
  npm trust 설정 후 실제 publish 증명은 새 prerelease로 `mode=next` workflow
  rehearsal을 다시 실행하고, workflow mechanics만 확인할 때만 `dry_run=true`를
  사용합니다. Tag-triggered publish는 prerelease가 아닌 version 전용입니다.
- 전환 후 `publish_provenance`는 **Codexus 자체 릴리스에 한해** 충족됨(checker가 검사하는
  임의 서드파티 패키지에 대해선 여전히 `informationalUnknown` — 그게 맞음).
- registry 전파 처리 개선: dist-tag 검증 단계가 read-after-write 지연에 실패하지 말고
  backoff로 retry.
- 일반 checker에 대해 `npm_2fa_status` / `publish_provenance`는 `informationalUnknown`
  으로 문서화하되, Codexus 자체 파이프라인은 trusted publishing으로 provenance를
  해결한다고 명시.

참고: [npm trusted publishers](https://docs.npmjs.com/trusted-publishers/),
[npm trust CLI](https://docs.npmjs.com/cli/v11/commands/npm-trust/).
