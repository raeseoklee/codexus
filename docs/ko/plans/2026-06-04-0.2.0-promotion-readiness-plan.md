# 0.2.0 승격 준비 계획

[English](../../plans/2026-06-04-0.2.0-promotion-readiness-plan.md)

상태: 첫 stable-promotion slice 구현됨

Codexus는 `0.2.0`을 `0.1.9` 다음에 자동으로 오는 버전으로 취급하면 안 됩니다.
기존 JSON contract는 `0.2.0`을 Codexus가 하나 이상의 experimental evidence
surface를 stable contract로 승격하거나, 이미 frozen된 stable field를 의도적으로
변경하고 changelog에 알리는 지점으로 정의합니다.

이 계획은 일정 약속이 아니라 guardrail입니다. 가까운 시점에 `0.2.0`을 반드시 잘라야
한다는 뜻이 아닙니다. Maintainer가 `0.2.0`을 검토할 때 version number 감각이 아니라
stable-contract evidence에 근거하도록 만드는 장치입니다.

## 핵심 결론

1. `0.2.0`은 countdown이 아니라 contract milestone입니다.
2. 넓은 0.2/0.3 기능 전체를 완료해야 `0.2.0`을 낼 수 있는 것은 아닙니다.
3. `0.2.0`을 자르기 전에는 최소 하나의 low-risk experimental surface를 stable로
   승격하고 frozen field를 문서화해야 합니다. `repo check --gate`가 이제 첫 번째
   승격 surface입니다.
4. 준비된 surface가 없다면 `0.2.0`을 강제하지 말고 `0.1.10`, `0.1.11`처럼
   `0.1.x` line을 계속 진행합니다.
5. App-instance start/stop, live autopilot, active relay adapter, Desktop
   attachment, automatic injection, plugin always-on claim 같은 action surface는
   lifecycle과 authority evidence가 더 강해질 때까지 experimental/deferred로 둡니다.

## 구현된 audit surface

`cx contract check --json`은 구조화된 0.2.0 promotion audit를 보고합니다.

```bash
cx contract check --target 0.2.0 --json
cx contract check --target 0.2.0 --gate --json
```

기본 report는 informational입니다. `--gate`는 audit된 candidate 중 최소 하나가 실제로
stable이고 그 frozen field가 `docs/json-contract.md`에 문서화되기 전까지 실패합니다.
첫 hardening slice는 `repo check --gate`를 승격했으므로 이제 `--gate`는 통과할 수
있습니다. 다만 나머지 action surface는 계속 experimental/deferred로 보고됩니다.

## 후보 집합

첫 audit slice는 다음 후보를 식별합니다.

- `repo check --gate`: 승격되었습니다. Local/static이고, repository knowledge gate로
  계속 dogfood 중이며, 외부 tool을 실행하지 않고, `docs/json-contract.md`에
  top-level field가 frozen되어 있습니다.
- `release check --gate`: local mode가 승격되었습니다. Static source-tree evidence로
  trusted publishing과 installer integrity를 보호하며, local-mode top-level field가
  frozen되어 있습니다. `release check --live`는 계속 명시적 experimental
  post-publish sign-off입니다.
- `lsp check --gate`: 명시적 bounded diagnostics command 실행에 한해 승격되었습니다.
  Protocol server를 시작하지 않고, unavailable/passed/failed branch와
  timeout/output-tail limit를 보고합니다. Protocol-server lifecycle과 자동 project LSP
  적용은 계속 deferred입니다.
- `architecture check --gate`: 좁은 declared-policy forbidden-import subset에 한해
  승격되었습니다. Broad layering은 계속 deferred입니다.
- `wiki context --fresh-only --gate`: 명시적 manual context selection에 한해
  승격되었습니다. Freshness gate는 local/structural이지만, automatic context injection은
  계속 deferred이며 이 surface와 함께 승격하면 안 됩니다.

## 명시적 non-candidate

첫 `0.2.0` promotion에는 다음을 포함하지 않습니다.

- `app instance start|stop`
- app-instance health modeling
- live `autopilot run`
- active multi-engine relay adapter
- Desktop app-server live attachment
- automatic context injection
- LSP protocol-server lifecycle과 자동 project LSP 적용
- plugin always-on supervision
- plugin-packaging always-on claim

이 surface들은 실제 process/control side effect가 있거나, upstream supported bridge가
없거나, distribution/discoverability experiment에 해당합니다.

## 승격 절차

1. 후보 하나 또는 둘을 고릅니다. 우선순위는 `repo check`와 local-mode `release
   check`입니다.
2. `docs/json-contract.md`와 Korean counterpart에 named top-level JSON field를
   freeze합니다.
3. Unknown additive field는 허용하되 named field를 제거하거나 재정의하지 않는다는
   contract test를 추가합니다.
4. 승격된 command에 대해 installed-package smoke를 추가합니다.
5. 구현 상태, 남은 작업, 칸반을 갱신해 승격된 surface는 stable로, 승격되지 않은
   surface는 experimental/deferred로 유지합니다.
6. 의도적으로 `0.2.0` cut을 평가할 때
   `cx contract check --target 0.2.0 --gate --json`을 실행합니다. Stable promotion이
   실제로 존재할 때만 통과해야 합니다.
7. `0.2.0` release note에는 승격된 surface를 명시하고, 어떤 0.2/0.3 track이 계속
   experimental인지 함께 적습니다.

## 현재 결과

0.2 promotion-hardening slice 이후 Codexus는 다섯 개의 stable 승격 surface를 가집니다:
`repo check --gate`, local-mode `release check --gate`, `lsp check --gate`, 좁은
`architecture check --gate`, manual `wiki context --fresh-only --gate`.
`cx contract check --target 0.2.0 --gate --json`은 stable promotion requirement를
통과할 수 있습니다. 그래도 `0.2.0` cut은 자동 다음 단계가 아니라 제품 결정입니다.
Action surface는 추가 hardening 또는 authority evidence가 필요합니다.
